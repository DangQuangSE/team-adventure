# ADR-003: Real-Time and Media Signaling - Socket.io and Mesh P2P WebRTC

**Date:** 2026-06-19
**Status:** Accepted

## Context

The Virtual Office MVP requires low-latency real-time video/audio calling and fast coordinate updates. Video calling requires high bandwidth, which is expensive to route through a central backend server. Coordination of connections (exchanging network addresses and codec details) requires a bi-directional signaling channel.

## Decision

We will use WebSockets via Socket.io for low-latency state synchronization and WebRTC signaling, while using a P2P Mesh network model for the actual video/audio transmission. Clients connect directly to one another to stream media, bypassing the Node.js server entirely.

## Consequences

### Positive
- Server bandwidth savings: Node.js server only routes light text messages (ICE candidates and coordinate strings), reducing hosting expenses.
- Ultra-low latency: Direct P2P media connections bypass server bottlenecks.
- High scalability for small teams (under 10 connections).

### Negative / Trade-offs
- CPU overhead: P2P Mesh scales as $O(N^2)$ connections, increasing client CPU usage as room density increases.
- NAT traversal issues: Connection success rates depend heavily on STUN servers and network firewall rules.

### Risks
- Large meetings causing client lag.
  * *Mitigation:* Limit private meeting room capacity to 8 players, and automatically disconnect non-proximity clients in public corridors.
