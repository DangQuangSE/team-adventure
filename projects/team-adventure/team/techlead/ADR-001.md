# ADR-001: Architectural Choice - Clean Architecture with Monolithic Deployment

**Date:** 2026-06-19
**Status:** Accepted

## Context

The Virtual Office MVP requires rendering a real-time multiplayer 2D grid and broker high-frequency signaling data. The developer team level is set to `mid`. We need to design a system that keeps the complex spatial coordinates calculation and business rules (e.g. proximity bounds and room permissions) decoupled from Express and Socket.io web technologies so we can test the logic independently.

## Decision

We will implement a Clean Architecture pattern packaged inside a monolithic deployment. The system will separate the codebase into distinct layers:
1. **Domain Entities:** Ephemeral representation of players, notes, and rooms.
2. **Use Cases:** Core application logic (e.g. spatial checks, room entry validations).
3. **Adapters:** Translates socket inputs and HTTP requests to the use cases, and vice versa.
4. **Frameworks & Drivers:** Express.js and Socket.io instances.

## Consequences

### Positive
- Framework independence: Domain logic is isolated and does not depend directly on Socket.io.
- High testability: We can run unit tests on spatial calculations without mocking socket channels.
- Simple deployment: Monolith deployment keeps build pipelines simple.

### Negative / Trade-offs
- Increased directory depth and file count compared to a simple flat structure.
- Higher conceptual barrier for junior developers joining the team.

### Risks
- Over-abstraction if developers write unnecessary interfaces for simple features.
  * *Mitigation:* Document the boundaries clearly and avoid boilerplate folders unless they serve a decoupling purpose.
