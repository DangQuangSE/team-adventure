# ADR-002: State Management - Ephemeral In-Memory Repositories

**Date:** 2026-06-19
**Status:** Accepted

## Context

The system processes real-time player coordinates (updated every 45ms per active player). Writing these coordinates immediately to a disk-based storage system (such as PostgreSQL) introduces significant I/O overhead and database thread locking risks, degrading multiplayer performance. Furthermore, since the workspace coordinates and active user connections are ephemeral and lose meaning after players close their browsers, permanent storage is not required for the initial MVP.

## Decision

We will implement in-memory state repositories inside the Node.js process using basic javascript objects (key-value maps). Use case layers will read and update this store via repository interfaces.

## Consequences

### Positive
- Maximum performance: State updates occur in CPU memory, yielding microsecond latency.
- Cost reduction: No additional managed database instances are required for staging/hosting.
- Simpler test mocking.

### Negative / Trade-offs
- No state persistence: Restarting the server drops all player states, active note content, and rooms.
- Memory scaling limits: The server process memory scales with player connection count.

### Risks
- Loss of collaborative notes on server reboot.
  * *Mitigation:* In the future, write a background job that periodically flushes notes to a flat JSON file on disk or Redis cache.
